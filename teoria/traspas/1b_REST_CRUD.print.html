Recapitulando: el API de una aplicacin web “clsica”

  • Las llamadas al API suelen ser verbos, que indican acciones: ponerNota, listarAlumnos
  • Los datos se envan en formato “nombre=valor&nombre2=valor2…”: orden=asc&convoc=ene2016

Adems y centrndonos en el cliente, las apps clsicas suelen usar plantillas generadas desde el servidor. El cliente se limita a renderizar HTML/CSS ya generado.

Aplicaciones REST

  • REST es en principio un “estilo”/arquitectura para el diseo de sistemas hipermedia. Es una serie de reglas de diseo de cmo interactuar con la aplicacin, es decir, reglas de diseo del API.

Por qu REST?

Sean por las razones que sean, en la actualidad la inmensa mayora de los APIs web siguen los principios de REST

REST bsico

  • REST puede verse como un intento de volver a los principios de HTTP. En teora REST es independiente de HTTP (pero entre nosotros, y sin que nos oiga Roy Fielding, son casi lo mismo)
  • Cambio de filosofa sobre las apps web clsicas: en lugar de disear mtodos concretos (ponerNota, listarAlumnos) aplicaremos una serie de operaciones ya predefinidas (GET, POST, PUT, DELETE) sobre los recursos (entidades) de nuestra aplicacin (asignaturas, alumnos, notas)

Recursos en REST

  • Un recurso es una entidad o una coleccin de entidades
  • Cada recurso se debe identificar con una URL nica

Tpicamente, las colecciones se ponen en plural, y un recurso concreto dentro de una coleccin se representa concatenando un identificador a la coleccin

Ejemplo de un supuesto campus virtual de la UA

/* todas las carreras */
http://api.cv.ua.es/carreras
/* Grado en Ingeniera Informtica */
http://api.cv.ua.es/carreras/GII/
/* todos los alumnos de la UA */
http://api.cv.ua.es/alumnos
/* Un alumno concreto */
http://api.cv.ua.es/alumnos/11222333N

IMPORTANTE: una URL de un recurso no es nada ms que un “puntero” al recurso. Para hacer algo til en el API habr que lanzar una peticin HTTP sobre esta URL

Operaciones en REST

Los mtodos POST/GET/PUT/DELETE se asocian respectivamente con las operaciones Crear/Leer/Actualizar/Borrar (CRUD), respectivamente

Ejemplos de “llamadas” al API: operacin + recurso

  • Obtener el listado de todos los alumnos: peticin HTTP
    • de tipo GET
    • a la URL http://miaplicacion.com/alumnos
  • Obtener los datos del alumno con DNI 11222333N: peticin HTTP
    • de tipo GET
    • a la URL http://miaplicacion.com/alumnos/11222333N
  • Dar de alta un alumno: peticin HTTP
    • de tipo POST
    • adjuntando los datos del nuevo alumno
    • a la URL http://miaplicacion.com/alumnos

Solo nos falta un formato estandarizado para los datos

  • Que el servidor enva al cliente en peticiones de tipo GET
  • Que el cliente enva al servidor en peticiones de tipo POST/PUT

Intercambio de datos en REST

  • JSON: formato estndar e independiente del lenguaje para representar objetos. Originario de Javascript
  • Es mucho ms legible y flexible que el nombre=valor&nombre2=valor2.... Se pueden representar objetos compuestos de otros, arrays, …

{
nombre: "Pepe Prez",
edad: 19, 
direccion: {
calle: "Pez",
num: 15}
}

En realidad REST no es ms que un conjunto de convenciones arbitrarias, que podran ser de otro modo

Por qu REST en lugar de otras alternativas? (II)

Para disear un buen API para los servicios necesitamos usar algo que la gente conozca. As que, aunque no hay nada superior desde el punto de vista tcnico en REST y JSON con respecto a usar RPC con un protocolo de ms bajo nivel, usar algo que la gente comprenda bien […] ayuda mucho en el diseo del API

Jay Krebs, [Lessons from Building and Scaling LinkedIn](http://www.infoq.com/presentations/linkedin-architecture-stack) (QCon, NY 2013)
